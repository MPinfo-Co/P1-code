# MP-Box 系統入門導覽

> 這份文件是給剛接觸這個專案、對前後端和資料庫都不熟的人看的。
> 從最大的圖開始，一層一層往下拆解。

---

## 第一課：整個系統長什麼樣？

想像一間餐廳：

```
顧客（資安專家）
    ↓ 進門點餐
【前台（Frontend）】  ← 你看到的網頁介面
    ↓ 傳訂單給廚房
【廚房（Backend）】   ← 真正處理邏輯的地方
    ↓ 需要食材時去倉庫拿
【倉庫（Database）】  ← 存所有資料的地方
```

你的系統也是這三層：

```
瀏覽器 http://localhost:5173     ← 前台（React）
    ↕ HTTP 請求
伺服器 http://localhost:8000     ← 廚房（FastAPI Python）
    ↕ SQL 查詢
PostgreSQL 資料庫                ← 倉庫（mpbox DB）
```

另外還有一個「後台自動廚師」：

```
Celery（排程工人）  ← 每 10 分鐘自動跑，不需要人觸發
    ↓
Claude AI（Haiku + Sonnet）
    ↓
PostgreSQL
```

## 第三課：後端 `backend/app/` 詳解

這是系統的心臟。分成幾個區域：

### `main.py` — 後端的大門

就像餐廳門口的指示牌，告訴你「登入請走 A 門、查事件請走 B 門」。

```python
app = FastAPI(...)           # 建立 API 伺服器
app.include_router(auth)     # 掛上「登入相關」路由
app.include_router(events)   # 掛上「事件相關」路由
```

---

### `api/` — 對外的窗口（API 端點）

| 檔案 | 負責什麼 | 對應網址 |
|------|----------|---------|
| `auth.py` | 登入、登出、拿 JWT token | `POST /api/auth/login` |
| `events.py` | 查詢/更新資安事件 | `GET /api/events` |
| `health.py` | 確認伺服器還活著 | `GET /health` |

**什麼是 API？** 前端說「給我事件清單」，後端回「好，這是 JSON 資料」。這個對話就叫 API 呼叫。

---

### `models/` — 資料庫的藍圖

告訴 PostgreSQL「請幫我建這些表格，每個欄位長這樣」。共 3 個檔案、9 個 class、9 張表格。

**`security_event.py`**

| class 名稱 | 對應 DB 表格 | 存什麼 |
|-----------|-------------|--------|
| `LogBatch` | `log_batches` | 每次從 SSB 拉 log 的紀錄 |
| `FlashResult` | `flash_results` | Haiku AI 分析每個 chunk 的結果 |
| `SecurityEvent` | `security_events` | 最終整理好的資安事件 |
| `DailyAnalysis` | `daily_analysis` | 每天 Sonnet 彙整的執行紀錄 |
| `EventHistory` | `event_history` | 人員對事件做了什麼處置 |

**`user.py`**

| class 名稱 | 對應 DB 表格 | 存什麼 |
|-----------|-------------|--------|
| `User` | `users` | 帳號資料 |
| `Role` | `roles` | 角色與權限設定 |
| `UserRole` | `user_roles` | 哪個帳號有哪個角色（關聯表） |

**`token_blacklist.py`**

| class 名稱 | 對應 DB 表格 | 存什麼 |
|-----------|-------------|--------|
| `TokenBlacklist` | `token_blacklist` | 登出後作廢的 JWT token |

---

### `schemas/` — 資料的「合約」

`models` 是給資料庫看的，`schemas` 是給 HTTP 請求/回應用的。

**比喻：** `models` 是倉庫的貨架格式，`schemas` 是發貨單的格式。

```
前端送來 JSON → schema 驗證格式對不對 → 寫入 DB（用 model）
DB 讀出資料（model）→ schema 轉換成 JSON → 回傳給前端
```

---

### `services/` — 對外溝通的專家

| 檔案 | 功能 |
|------|------|
| `ssb_client.py` | 負責連到 192.168.10.48 的 SSB 設備，拉 log |
| `claude_flash.py` | 負責呼叫 Claude Haiku（快速分析） |
| `claude_pro.py` | 負責呼叫 Claude Sonnet（深度彙整） |

這三個就是「對外聯絡窗口」，其他程式想拉 log 或問 AI，就呼叫這裡的函數。

---

### `tasks/` — 自動排程工作

| 檔案 | 執行時機 | 做什麼 |
|------|----------|--------|
| `flash_task.py` | 每 10 分鐘 | 拉 log → 送 Haiku → 產 group_key → 合併 |
| `pro_task.py` | 每天 02:00 | 讀累計結果 → 送 Sonnet → 寫入正式事件 |

---

### `worker.py` — 排程鬧鐘

這個檔案只是設定「什麼時候叫誰去跑」，實際工作在 `tasks/` 裡。

```python
"flash-task": 每 10 分鐘執行 flash_task
"pro-task":   每天 02:00 執行 pro_task
```

---

### `core/` — 基礎設施

| 檔案 | 功能 |
|------|------|
| `config.py` | 讀取 `.env` 裡的所有設定值（密碼、API key 等） |
| `security.py` | JWT token 的產生與驗證（登入用） |
| `deps.py` | 「這個 API 需要登入才能用」的驗證邏輯 |

---

### `db/session.py` — 資料庫連線

建立連到 PostgreSQL 的連線。其他程式需要查資料時，從這裡拿連線。

---

### `alembic/` — 資料庫版本控制

**比喻：** Git 管理程式碼版本，Alembic 管理資料庫結構版本。

```
alembic/versions/
├── 5db1ff... → 建立 users, roles 表格
├── 4ccbef... → 建立 security_events 等 5 個表格
└── 16f46c... → 建立 token_blacklist 表格
```

執行 `alembic upgrade head` = 把所有版本跑完，DB 就有完整的表格。

## 第四課：前端 `frontend-mui/src/` 詳解

React 的概念：每個 `.jsx` 檔案就是一個「積木（Component）」，組合起來變成畫面。

### 整體結構

```
src/
├── main.jsx          ← 程式最入口，把 App 掛到 HTML
├── App.jsx           ← 定義「哪個網址顯示哪個頁面」
├── theme.js          ← 整個網站的顏色/字型設定
│
├── pages/            ← 每個頁面
├── components/       ← 可重複使用的積木
├── contexts/         ← 跨頁面共享的狀態
└── data/             ← 假資料（mock data）
```

---

### `App.jsx` — 路由地圖

```
/login                       → 登入頁
/                            → 首頁
/ai-partner                  → AI 夥伴列表
/ai-partner/xxx/issues       → 事件清單
/ai-partner/xxx/issues/yyy   → 事件詳情
/kb                          → 知識庫
/settings/...                → 設定頁
```

---

### `pages/` — 各頁面

| 資料夾 | 頁面 | 功能 |
|--------|------|------|
| `Login/` | 登入頁 | 帳密驗證 |
| `AiPartner/IssueList.jsx` | 事件清單 | 顯示所有資安事件，有篩選/分頁 |
| `AiPartner/IssueDetail.jsx` | 事件詳情 | 3 個 Tab：詳情/歷史/處置紀錄 |
| `KnowledgeBase/` | 知識庫 | 文件查詢介面 |
| `Settings/` | 設定 | 帳號/角色/AI 設定 |

---

### `components/Layout/` — 共用外框

每個頁面都有同樣的外框（側邊欄 + 上方 Header）。

```
┌──────────┬───────────────────────┐
│ Sidebar  │  Header               │
│          ├───────────────────────┤
│ 側邊選單  │  <頁面內容在這裡>      │
└──────────┴───────────────────────┘
```

---

### `contexts/` — 跨頁面共享資料

**問題：** A 頁面登入了，B 頁面怎麼知道？
**解法：** 用 Context，像廣播站一樣，所有頁面都能收到。

| 檔案 | 廣播什麼 |
|------|---------|
| `AuthContext.jsx` | 目前登入的使用者是誰、JWT token |
| `IssuesContext.jsx` | 事件清單資料（從後端 API 拿回來後共享） |

---

### `data/` — 假資料

在後端 API 還沒接好時，先用假資料讓前端跑起來。

| 檔案 | 內容 |
|------|------|
| `users.js` | 帳號清單（Rex 等，密碼 123） |
| `issues.js` | 假的資安事件資料 |
| `aiPartners.js` | AI 夥伴的設定資料 |

## 第五課：資料怎麼流動？

### 情境 A：資安專家查看事件清單

```
1. 開瀏覽器 → http://localhost:5173/ai-partner/xxx/issues
2. React IssueList.jsx 發出請求：GET http://localhost:8000/api/events
3. FastAPI events.py 收到請求 → 查詢 PostgreSQL security_events 表格
4. 回傳 JSON 資料給前端
5. IssueList.jsx 把資料渲染成表格顯示
```

---

### 情境 B：每 10 分鐘自動分析 log

```
1. Celery Beat 鬧鐘響 → 觸發 flash_task.py
2. ssb_client.py 連到 192.168.10.48 拉最新 1,500 筆 log
3. 精簡 log（只留 5 個欄位，省 65% token）
4. 切成 5 個 chunk，每 chunk 300 筆
5. 逐一送給 claude_flash.py → 呼叫 Claude Haiku 分析
6. 程式用 group_key 邏輯去重合併
7. 存入 flash_results 表格
```

---

### 情境 C：每天 02:00 最終彙整

```
1. Celery Beat → 觸發 pro_task.py
2. 讀 flash_results 裡 chunk_index=-1（當天累計）
3. 送給 claude_pro.py → 呼叫 Claude Sonnet 深度分析
4. 寫入 security_events 表格
   ├── 若 match_key 相同且未結案 → UPDATE（合併延續）
   └── 若新的事件 → INSERT
```

## 第六課：幾個關鍵概念

### JWT Token（登入認證）

登入時後端產一個「通行證」（JWT）給前端，之後每次打 API 都夾帶這個通行證，後端確認合法才給資料。存在瀏覽器的 `localStorage`。

### group_key（事件去重的核心）

同樣的攻擊行為，可能產生幾百條 log。用 group_key 把它們合併成「同一個事件」，避免資安專家看到一堆重複通知。

| 類型 | group_key 範例 | 合併範圍 |
|------|--------------|---------|
| 外部來源被 deny | `deny_external_211.21.43` | 同一目標網段 |
| 內部來源被 deny | `deny_internal_192.168.10.4` | 同一內部 IP |
| Windows 登入失敗 | `4625_administrator` | 同帳號所有失敗 |
| Windows 帳號鎖定 | `4740_mp0263` | 同帳號所有鎖定 |

### Celery + Redis（排程引擎）

- **Redis**：工作佈告欄，Celery Beat 把「要執行的任務」貼上去
- **Celery Worker**：看著佈告欄，有任務就去執行
- **Celery Beat**：定時把任務貼上佈告欄的人

### alembic migration vs seed.py

- `alembic`：建立表格結構（空的架子）
- `seed.py`：塞入初始資料（預設帳號等）

## 第七課：你不常改但需要知道的檔案

| 檔案 | 為什麼存在 |
|------|-----------|
| `docker-compose.yml` | 一鍵啟動 PostgreSQL + Redis + 後端 + 前端 + Celery |
| `backend/.env` | 密碼、API key，**不進 git** |
| `backend/.env.example` | `.env` 的範本，讓人知道需要設定哪些 |
| `backend/requirements.txt` | Python 套件清單（`pip install -r` 這個） |
| `frontend-mui/package.json` | JS 套件清單（`npm install` 這個） |
| `frontend-mui/vite.config.js` | 把前端的 `/api` 請求轉到 port 8000 |
| `backend/scripts/run_pipeline.py` | 手動跑一次完整管線（測試用，不用等排程） |

## 總結：你上班時要改的地方

| 任務 | 改哪裡 |
|------|--------|
| 修改畫面顯示 | `frontend-mui/src/pages/` |
| 修改 API 回傳格式 | `backend/app/api/` + `schemas/` |
| 修改 AI 分析邏輯 | `backend/app/services/claude_*.py` |
| 修改事件合併規則 | `backend/app/tasks/flash_task.py` |
| 新增資料庫欄位 | `backend/app/models/` → 建新 alembic migration |
| 修改排程時間 | `backend/app/worker.py` |
